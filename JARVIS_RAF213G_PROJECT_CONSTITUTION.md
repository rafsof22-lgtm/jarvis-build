# JARVIS / RAF213G Project Constitution

**Version:** 1.1.0  
**Effective:** 2026-07-22  
**Canonical repository:** `rafsof22-lgtm/jarvis-build`

## Governing objective

Build Jarvis / RAF213G as an evidence-led, modular and reversible operating system. Preserve user intent, source truth, safety boundaries, service isolation, cost control, auditability and rollback.

## Mandatory startup

Before material project work, read:

1. `PROJECT_CONSTITUTION.md`
2. this constitution
3. `PROJECT_CONTINUITY.md`
4. `AGENTS.md`
5. applicable module-local instructions, deployment ledger and rollback file
6. current issue, branch, pull request and runtime evidence

These files are available only to chats or agents with this repository or project connected. They are not account-wide hidden memory.

## Source authority

Use this order when sources conflict:

1. current explicit user instruction;
2. this constitution and continuity file;
3. current merged repository truth;
4. authorised runtime/provider evidence;
5. user-provided primary files and exact exports;
6. current official external sources;
7. historical summaries and inferred context.

Preserve contradictions and supersession links.

## Automatic Jarvis relevance intake

Any accessible chat, export message, uploaded file, repository item, issue, pull request, attachment, transcript or source pack that materially mentions or implies Jarvis, RAF213G, an agent, framework, module, workflow, runtime, Command Centre, intelligence gathering, source reconstruction, deployment or an existing Jarvis specialist domain is a **candidate build input**.

For each candidate:

1. preserve the raw source or immutable pointer;
2. record provenance, timestamp, hash where available and access boundary;
3. capture both user requests and assistant responses when present;
4. classify chronology, project, domain, module, agent, integration, requirement and open loop;
5. deduplicate only true duplicates while preserving every source lineage pointer;
6. place each unique requirement at the narrowest correct reusable layer;
7. map it to artifact, implementation, test/waiver, evidence, runtime state and rollback;
8. mark inaccessible content `PENDING_INGEST` or `BLOCKED_BY_ACCESS` rather than claiming capture.

A mention alone does not prove applicability. Incidental references must be excluded with a recorded reason. No chat is considered integrated merely because its title or URL is registered.

## Chronology and consolidation law

- Maintain one canonical chronological source ledger and one consolidated priority tracker.
- Preserve historical trackers as immutable lineage; new trackers supersede but do not erase them.
- Sort by source event time where available, then ingestion time, while recording clock uncertainty.
- Keep raw sources separate from normalized requirements and implementation artifacts.
- Cross-link requirements that apply to multiple modules instead of duplicating their canonical text.
- Preserve conflicting instructions as separate records until a DecisionRecord resolves precedence.
- Repository, project and module trackers must reconcile to the canonical programme tracker.

## Requirement preservation

- Preserve unique requirements, wording, boundaries and acceptance gates.
- Do not silently delete, weaken or overwrite prior requirements.
- Record deprecation with reason, owner, replacement, evidence and rollback.
- Maintain stable requirement, module, capability, artifact and evidence identifiers.
- Maintain inaccessible-source and unresolved-gap ledgers.

## Completion contract

A task is complete only when this chain is satisfied or explicitly waived:

`Requirement -> Module -> Artifact -> Test or waiver -> Evidence -> Runtime state -> Rollback -> Owner acceptance`

Do not claim total completion, production readiness or end-to-end verification from source code, CI or a narrow route alone.

## Proof states

Use `PROVEN`, `NEEDS_PROOF`, `ASSUMPTION`, `GAP`, `RISK`, `OPEN_LOOP`, `BLOCKED_AT_EXACT_STEP`, `PENDING_INGEST`, `BLOCKED_BY_ACCESS`, `VISIBLE_SCOPE_VERIFIED` and `END_TO_END_NOT_VERIFIED` consistently.

## Change control

- GitHub is source truth for code, governance, schemas, tests, issue state and rollback history.
- Prefer small branches and pull requests.
- Inspect existing files before editing.
- Preserve a working baseline and make atomic changes.
- Run deterministic tests and security/history scans.
- Merge only after required checks pass.
- Record changed files, risks, gaps, runtime impact, rollback and next action.
- Never rewrite history without explicit approval.
- Do not delete repositories, branches, files, issues, PR history or source records merely to reduce duplication; quarantine, supersede or archive them with evidence first.

## GitHub and pull-request reconciliation

For every accessible Jarvis-related repository:

- inventory repository purpose, owner, visibility, default branch, current head and deployment mapping;
- classify duplicate capabilities, abandoned scaffolds, generated artifacts, secrets risk and cross-repository ownership;
- reconcile open and closed pull requests, review threads, CI status, merge evidence and supersession;
- keep independently deployable services isolated;
- use branch -> tests/scans -> PR -> independent verification -> merge -> staging -> proof;
- require explicit approval before destructive cleanup, history rewrite, repository deletion, force-push or production-impacting changes.

## Approval boundaries

Explicit approval is required before production-impacting actions, provider or billing changes, credential creation or rotation, destructive data operations, paid services, external communications, financial execution or clinical/device actions.

Safe repository changes, tests, reports, placeholders and staging plans may proceed when clearly authorised.

## Secrets

Never commit or request real passwords, tokens, cookies, private keys, recovery material or production credentials in chat. Store secret values only in approved provider variables, GitHub secrets or a vault. Repository files contain names and placeholders only.

## Service isolation

A shared repository does not imply shared runtime. Each module retains its own service root, deployment mapping, variables, permissions, data stores, routes, tests, evidence, backup, restore and rollback. Do not repurpose another module's runtime to make a different module appear complete.

## Domain reconstruction boundaries

- XRP/HBAR and selected asset intelligence remains research-only and evidence-ranked. Update controls must define source budgets, source classes, freshness, deduplication, probability methodology, discrepancy logs and anti-hype checks; a requested 50–100-source scan is a target, not proof of coverage.
- Tax and Active Trust are separate legal/tax batches even when they share source buckets. Current conclusions require official Australian sources and professional-review gates.
- Finance Planning and Financial New map into isolated AI CFO requirement/evidence registries; accounting credentials and production records remain segregated.
- Longevity and Health reconstruction remains privacy-safe, restricted, non-diagnostic and evidence-tiered.
- Bill CFO, Jarvis Health, XRP/HBAR Apex, VTI, Hub and Property retain independent service roots, credentials, databases, tests, deployments and rollback.

## Knowledge and provenance

Keep raw evidence separate from summaries. Record source identity, authority, hash where possible, coordinates, parser version and access limits. Enforce permissions, freshness, duplicate suppression, contradiction handling and citation coordinates. Never claim inaccessible files were processed.

## GitHub capability intake

External repositories begin as untrusted. Use `skills/github-repo-capability-scout/` for discovery, canonical-link resolution, provenance and licence review, static quarantine, capability extraction, free-first comparison, duplicate analysis, Jarvis placement, approval, sandbox tests, staging, verification, rollback and registry updates.

Static analysis is not proof that software is harmless. High-risk candidates remain blocked until isolated review.

## Free-first governance

Compare existing capability, deterministic local implementation, free/open-source/self-hosted, official free tier, cheapest viable paid, best-value managed and premium fallback only when justified. Track setup, reliability, scale, maintenance, quotas, lock-in, security and true task cost.

## Domain boundaries

Health-related modules remain privacy-safe, evidence-led and non-diagnostic. Financial and market modules remain research-only and paper-first unless a separately authorised and governed runtime exists. Property workflows remain capture, staging and approval-controlled promotion by default.

## Runtime verification order

Verify source/configuration, build/CI, deployment and commit, health, readiness, status, provider authentication, smallest real route, persistence, retries/idempotency, backup/restore, rollback and finally end-to-end workflow. A narrow success does not prove broader readiness.

## External staging and production gate

Do not connect Command Centre or the 18-layer digital twin to external services until staging endpoints, IAM, service identities, secret references, databases, budgets/cost ceilings, observability, backups, rollback ownership and acceptance tests are approved and recorded. Production promotion requires separate owner acceptance.

## Priority execution

Use `PROJECT_CONTINUITY.md`, `docs/full-completion-priority-plan.md`, `registry/full_completion_priority_plan_v1.json` and the latest `registry/trackers/all_progress_tracker_reconciliation_v*.json` as the current queue. Execute the highest-priority safe task and stop only at the exact remaining external or approval gate.

## Amendment

Amend through a branch and pull request, preserve prior versions in Git, document rationale, run constitution/continuity validation and obtain owner approval for material boundary changes.
