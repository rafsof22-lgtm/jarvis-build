# Verified Jarvis Module Handoff Standard

Status: READY_FOR_REVIEW

## Scope

Apply this standard to the active XRP/HBAR Framework Options module and to future independently usable Jarvis modules when their source material is accessible.

Do not restart, simplify, overwrite, compress away or remove approved requirements. Preserve raw sources, exact pointers, hashes, versions, amendments, superseded history and unresolved conflicts.

## Source accounting

Create and maintain a known source denominator covering visible user requests, adopted assistant responses, uploaded/project files, pages, paragraphs, sentences, tables, spreadsheet rows, repositories, code, configurations, workflows, registries, deployments, runtime evidence and authorised connectors, tools, APIs, Skills, plugins and MCPs.

Record inaccessible, failed, excluded and pending sources. Never silently omit them.

## Required extraction

Extract and reconcile goals, requirements, instructions, modules, agents, features, strategies, frameworks, workflows, tools, APIs, connectors, Skills, plugins, models, inputs, outputs, schemas, memory, data, credential names and scopes only, security controls, approvals, costs, dependencies, limits, tests, evidence, deployment, monitoring, rollback, risks, gaps, conflicts, decisions and OpenLoops.

## Required verification sequence

Run source inventory, extraction, reconstruction, reconciliation, consolidation, delta scan, duplicate detection, conflict/discrepancy detection, gap analysis, applicability review, traceability checking, specification-versus-implementation comparison, security review, cost review, testing and validation, rollback review and no-gaps verification.

Do not claim completion without:

`Source -> Request -> Requirement -> Module -> Artifact -> Test/Waiver -> Evidence -> Deployment -> Rollback`

## Independent module and federation rule

Keep each module independently usable. Do not flatten, destructively merge or compress module repositories into `jarvis-build`. Integrate through versioned APIs, events, MCPs, queues, schemas and contracts. Preserve module-specific source history, secrets, deployment, monitoring and rollback.

## Required Jarvis Module Handoff Pack

The pack must include module ID, name, owner, purpose, canonical requirements, source and coverage manifest, extraction ledger, delta/duplicate/conflict/discrepancy reports, architecture and interfaces, agents, Skills, tools, models, APIs, MCPs, connectors, events, inputs, outputs, schemas, data stores, memory, credential names and scopes only, workflows, triggers, dependencies, versions, implementation paths, tests, evidence, deployment, health checks, security, approvals, costs, limits, rollback, disaster recovery, unresolved risks, gaps, conflicts, Jarvis placement, federation plan, readiness percentage and exact next actions.

Do not mark the handoff VERIFIED unless the source pack, requirements manifest, tests and evidence all exist.

## External statuses

Use only:

- PENDING
- PARTIAL
- READY_FOR_REVIEW
- VERIFIED
- BLOCKED

Map detailed runtime states into these five external statuses without losing internal evidence.

## Response and tracker rule

At the bottom of every substantive module response, show the complete visual tracker using:

`| Workstream | Progress | Status | Evidence or blocker | Next action |`

Also show overall completion, verified completion, source-analysis coverage, requirements coverage, implementation, testing and verification, deployment readiness, security readiness, Jarvis integration readiness, unresolved gaps, blockers, owner actions and the next highest-priority safe action.

Preserve existing tracker rows, update only from evidence and add new rows in the correct group. Progress may decrease when verified scope expands. A row reaches 100% only with the full proof chain.

## Continuation and authority

Continue automatically from the highest-priority safe, unblocked incomplete item. Standing approval covers safe reversible no-new-cost work, branches, draft PRs, tests, scans, registries, documentation and staging preparation.

Pause for secret values, login, MFA, CAPTCHA, OAuth consent, billing/legal acceptance, public publishing, destructive production changes, production promotion, money movement, live trading or regulated decisions.

Save or update the full handoff pack with manifests, hashes, evidence, changelog, rollback and resume prompt whenever material work changes the module.
