# Jarvis Domain Federation and Consolidation Master Plan

**Date:** 20 July 2026 (Australia/Melbourne)  
**Owner:** Tas Raptis / RAF213G  
**Control plane:** `rafsof22-lgtm/jarvis-build`  
**Status:** `SPEC_ONLY` until every requirement has implementation, test or waiver, evidence and rollback mappings.

## 1. Executive decision

Jarvis will retain a detailed modular backend while presenting a small, novice-friendly owner interface.

The system must not delete or flatten specialist capabilities merely to make the interface simpler. It will use progressive disclosure:

- one owner-facing Command Centre page;
- seven primary workspaces;
- clear status, next action, cost, risk and approval cards;
- specialist modules, agents, skills, integrations, records and advanced parameters behind each workspace;
- shared governance, source truth, security, cost and evidence controls in the backend.

This consolidates navigation, not domain responsibility, evidence lineage, permissions or runtime isolation.

## 2. Source and truth boundary

The verified historical textual denominator currently includes 2,610 conversations, 357,835 message nodes and 53,473 user messages. Candidate requirement extraction has been performed, but the exact independently reviewed and semantically deduplicated requirement total is not yet proven.

Historical files that claim 100 percent completion, zero gaps, 99 percent accuracy, guaranteed passive income, guaranteed trading profitability or 500-plus operational modules are preserved as source claims only. They do not constitute implementation or deployment proof.

Use only these evidence-backed states:

- `PENDING_INGEST`
- `SPEC_ONLY`
- `BACKLOGGED`
- `SCAFFOLDED`
- `IMPLEMENTED_NOT_INTEGRATED`
- `INTEGRATED_STAGING`

- `BLOCKED`

Recommended chats:

- `JARVIS CONTROL CENTRE`
- `AUTONOMOUS REVENUE FACTORY`
- `TRADING AND MARKETS LAB`
- `BILL CFO FAMILY OFFICE`
- `JARVIS HEALTH`
- `XRP HBAR APEX INTELLIGENCE`
- `BUSINESS PROPERTY AND ASSETS`

Each chat must:

- route durable requirements to `jarvis-build`;
- preserve word-for-word request references where accessible;
- use the shared status vocabulary;
- publish structured events or handoff artifacts;
- keep sensitive finance and health data isolated;
- avoid copying entire private source corpora into public repositories;
- maintain module-local memory and permissions;
- expose blockers and exact next actions to Jarvis Control.

## 7. Shared architecture

`Owner Chat and One-Page Console -> Gateway -> Sovereign Control Plane -> Domain Router -> Domain Modules -> Skills and Tools -> Model and Cost Router -> Data and Evidence Stores -> Independent Verification -> Reports and Alerts`

Cross-domain communication uses versioned events and references rather than unrestricted shared memory.

Minimum event fields:

- event ID;
- schema version;
- producer;
- consumer;
- occurred time;
- correlation ID;
- idempotency key;
- source references;
- evidence references;
- approval state;
- data classification;
- cost record;
- retry count;
- governed status;
- structured payload.

## 8. Data isolation

Data classes:

- public research;
- internal business;
- confidential financial;
- tax and legal privileged-preparation material;
- health and sensitive personal information;
- credentials and authentication material;
- trading and portfolio records.

Rules:

- credentials never enter model memory;
- finance and health use separate storage boundaries and access policies;
- public `jarvis-build` stores schemas, status and sanitized evidence, not private raw records;
- every cross-domain read must be purpose-bound and logged;
- retention, deletion, legal hold, export and correction processes must be defined.

## 10. Maximum requirement extraction program

To reach an evidence-backed full historical specification:

1. recover the generated requirement-candidate register;
2. verify its source hash and row count;
3. classify candidates by instruction, goal, objective, requirement, feature, function, workflow, constraint, decision and implied control;
4. link every candidate to message occurrence IDs;
5. perform semantic deduplication while preserving duplicate lineage;
6. preserve conflicting variants and create Decision Records;
7. independently review candidates in bounded batches;
8. map accepted candidates to domains and modules;
9. map modules to repository artifacts and implementation states;
10. add tests or approved waivers;
11. attach evidence and rollback;
12. enumerate inaccessible binary objects and owner exclusions;
13. rerun no-gaps verification over the declared denominator.

Universal 100 percent completion remains `BLOCKED` until this chain is complete.

## 14. Current blockers

- exact deduplicated historical requirement count is not exposed and independently reviewed;
- many historical attachments are pointer-only and their bytes are inaccessible;
- private finance and health source sets require secure authorised ingestion;
- no dedicated Revenue Factory repository is currently proven;
- no dedicated Trading Lab repository is currently proven;
- Bill CFO runtime remains primarily an isolated service shell and source-intake capability;
- several services remain deployment-unverified;
- live trading requires a separate high-risk approval program;
- regulated tax, legal, financial and medical outputs require professional boundaries;
- provider credentials, billing and production exposure remain owner gates.

## 15. Completion rule

This master plan is complete as a specification artifact only. It is not proof that all domains are implemented or deployed.

A domain becomes `DONE_VERIFIED` only when:

`Requirement -> Module -> Artifact -> Test/Waiver -> Evidence -> Deployment or accepted non-deployment outcome -> Rollback`
