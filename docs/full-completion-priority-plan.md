# Full Completion Priority Plan

This is the canonical human-readable order for bringing the visible Jarvis workspace to full completion. The machine-readable source is `registry/full_completion_priority_plan_v1.json`.

## Execution principle

`P0 availability/security -> P1 shared runtime/persistence -> P2 source lock -> P3 specialist modules -> P4 staging federation -> P5 production -> P6 final reconciliation`

## P0 — Availability, security and provider proof

1. Hub DigitalOcean Recovery Console and route recovery.
2. VTI historical provider-token revocation and rotation evidence.
3. Jarvis Health authoritative Railway domain, privacy-safe runtime and rollback.
4. Property Apps Script webhook and one-row raw staging proof.
5. Fresh Hub deployment/Gmail proof workflow.
6. Protected Cobalt host, authorised canary and purge.
7. VTI-to-Hub signed staging federation canary.

## P1 — Shared runtime and persistence

1. Property Postgres, signed queue, n8n and approval-controlled promotion.
2. Connected Knowledge Fabric benchmark and remaining SOP/OJT packs.
3. Governed source-discovery/API intake.
4. VTI OAuth/export/evidence/worker backlog.
5. Command Centre phases 2–6.
6. Infrastructure, security, observability, backup, restore and cost guards.

## P2 — Exact source lock

1. Complete the accessible ChatGPT/archive denominator.
2. Complete Jarvis core requirement/provenance reconstruction.
3. Process XRP/HBAR reconciliation batch 002.
4. Complete Health/Spooky2 source and evidence-tier separation.
5. Build universal intelligence and opportunity research fabric.
6. Complete shared-core specialist workstreams.

## P3 — Specialist operating modules

Build operating domains, AI CFO, XRP/HBAR–CFO integration, paper-first trading, agency and product factories only after their source and shared-platform dependencies are stable.

## P4 — Staging and end-to-end proof

Prove contracts, authentication, privacy, negative tests, persistence, retries, idempotency, dead letters, backups, isolated restore, rollback, monitoring and cost limits across connected modules.

## P5 — Production promotion

Create bounded approval tickets for each production service. Record deploy SHA, domain, variable names, migrations, smoke evidence and rollback. Never reuse another module's production service.

## P6 — Final closure

Reconcile all open issues and requirements. Generate a release manifest and final Command Centre view. Conduct security/privacy/domain-boundary review, regression suite, restore drill and explicit owner acceptance.

## Completion contract

Every task requires:

`Requirement -> Module -> Artifact -> Test or waiver -> Evidence -> Runtime state -> Rollback -> Owner acceptance`
