# Jarvis Self-Evolution Control Plane and Unified Assistant v1

## Outcome

This tranche converts the self-coding, self-repair, editable-framework, manual-override, voice-control, and popup-assistant requirements into a dependency-free local runtime contract.

## Architecture

```text
User text or voice
        |
Unified Jarvis Pop Assistant
        |
Command Preview and Policy Gate
        |
Editable Object Registry
        |
Change Request -> Diff -> Simulation -> Tests -> Approval
        |
Reversible Staging Apply -> Independent Verification -> Rollback
```

## Editable surface

The registry covers frameworks, specifications, workflows, stacks, architecture, orchestrators, Skills, agents, prompts, instructions, policies, schemas, model routes, tools, integrations, connectors, UI, tests, deployment profiles, memory policies, sources, modules, services, SOPs, and OJT packs.

## Manual and automatic control

The user can view, compare, edit, pause, reject, approve, select a model, request more tests, or roll back. Jarvis can automatically repair only objects explicitly marked `AUTO_REVERSIBLE`, and only in staging after validators pass.

Policy, deployment, credential, financial, destructive, connector, and production changes remain gated. Voice follows the same restrictions as text.

## Jarvis Pop UX contract

The popup contract includes chat history, voice transcript, model selector, active-object context, object browser, source and evidence panel, diff preview, approval queue, activity timeline, health and tests, rollback history, suggested prompts, and accessible mobile controls.

## Source consolidation truth

Current repository and supplied-source work is consolidated only within verified bounded denominators. Missing exports, incomplete workspace bytes, unavailable project chats, opaque archives, external provider state, and unmounted attachments remain `PENDING_INGEST` or `BLOCKED_BY_ACCESS`.

The presence of historical files claiming 100 percent does not establish complete reconstruction, live integration, production deployment, or owner acceptance.

## Status

The repository implementation may reach `DONE_VERIFIED` only for the local runtime and deterministic test scope after exact-head CI passes. Connected voice, LLM providers, live repositories, staging deployment, production and universal source coverage remain separate gates.
