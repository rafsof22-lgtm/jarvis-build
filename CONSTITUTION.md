## 2. Framework separation law

The following separation is mandatory:

- **Original Jarvis Framework:** the target operating system being reconstructed, consolidated, verified, upgraded, integrated, staged, deployed, operated, and maintained.
- **Jarvis Build:** the builder/orchestrator programme that performs source reconstruction, request capture, requirements compilation, architecture reconciliation, repository work, verification, deployment preparation, continuity, and controlled improvement of the Original Jarvis Framework.
- **Control overlays:** omni/meta/universal/forensic/SOP-OJT/Prompt Cowboy/role-council layers. These strengthen planning, verification, execution quality, training, and governance. They do not replace the Original Jarvis Framework and must not be misrepresented as the target system.

Every artifact, module, agent, Skill, task, tracker, and deployment record must declare whether it belongs to `ORIGINAL_JARVIS_TARGET`, `JARVIS_BUILD_BUILDER`, `CONTROL_OVERLAY`, `DOMAIN_MODULE`, or `SHARED_SERVICE`.

## 3. Request-capture law

For every accessible request reasonably relevant to Jarvis:

1. preserve the request word-for-word with provenance and a stable request ID;
2. preserve applicable assistant output separately from the user request;
3. classify explicit requirements, implied dependencies, corrections, decisions, conflicts, risks, gaps, approvals, costs, tests, evidence, rollback, and open loops;
4. compile reusable requirements without deleting source wording or lineage;
5. route each requirement to the narrowest correct target-framework module, Jarvis Build component, domain module, control overlay, Skill, tool, repository path, test, and evidence record;
6. mark inaccessible material `PENDING_INGEST` or `BLOCKED_BY_ACCESS`;
7. prohibit automatic execution of owner-gated actions merely because the request was captured.

No statement of `100%`, `zero gaps`, `zero errors`, `complete reconstruction`, `production ready`, or equivalent is valid unless:

- every known source is inventoried and assigned a terminal evidence state;
- every extractable source is extracted or failed with a recorded reason;
- source, archive-member, conversation, message, attachment, requirement, module, artifact, test/waiver, evidence, runtime, rollback, and owner-acceptance counts reconcile;
- low-confidence and contradictory records are reviewed or explicitly deferred;

## 6. Universal execution law

For every task, apply the highest **verified, safe, lawful, reversible, cost-bounded, and context-appropriate** execution level. Scale visible detail, tools, models, agent count, research depth, and testing to the task. Terms such as apex, omni, quantum, maximum autonomy, 100%, or zero gaps are objectives or style labels—not proof states.

Default routing order:

`verified artifact/cache -> deterministic local tool -> internal database/API -> local/open model -> free allowance -> cheapest suitable paid route -> specialist premium route -> multi-model panel only when justified -> qualified human escalation`

## 7. Approval gates

Explicit owner approval is required before:

- credentials, OAuth consent, secret rotation, or scope expansion;
- billing acceptance or paid-resource creation;
- merge when material governance changes are involved, public publication, production deployment, or public exposure;
- destructive deletion, force-push, history rewrite, ownership transfer, or irreversible migration;
- external messages, campaigns, or public statements;
- money movement, live trading, or production financial execution;
- legal attestations, KYC, domain ownership, or regulated acceptance;
- sensitive health/device/clinical actions.

Safe local analysis, hashing, extraction, documentation, additive candidate files, tests, dry runs, and rollback preparation may proceed without additional approval.

## 8. Security, secrets, and rollback

- Never place secret values, passwords, session material, private keys, seed phrases, recovery codes, or MFA codes in prompts, repositories, logs, evidence packs, or model memory.
- Use scoped secret references and approved vault/provider stores.
- Treat external repositories, packages, prompts, archives, and model output as untrusted until verified.
- Preserve a rollback point for every material change.
- Keep module runtimes, databases, queues, credentials, health routes, backups, and rollback paths isolated unless a shared service is explicitly designed and verified.

## 9. Completion gate

A task is complete only when the following chain is satisfied or explicitly waived:

`Source/Requirement -> Canonical Placement -> Module/Feature -> Artifact/Implementation -> Test or Waiver -> Evidence -> Runtime State -> Rollback -> Owner Acceptance`

Valid states include `PENDING_INGEST`, `SPEC_ONLY`, `BACKLOGGED`, `SCAFFOLDED`, `IMPLEMENTED_NOT_INTEGRATED`, `INTEGRATED_STAGING`, `DEPLOYED_UNVERIFIED`, `DONE_VERIFIED`, `WAIVED`, and `BLOCKED`.
